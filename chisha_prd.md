# 《吃啥嘞》产品需求文档（PRD）

## 项目名称

吃啥嘞（chisha.shameless.top）

一句话定义：

> 一个带有"抽签仪式感"的中原饮食随机选择器。

---

# 一、产品定位

## 产品目标

解决：

> "今天吃什么" 的决策疲劳。

但不仅仅是功能性解决。

产品核心价值：

- 提供轻量随机决策
- 提供情绪价值
- 提供仪式感
- 提供文化气质

本产品不追求：

- 功能复杂
- 平台化
- 社交化
- AI 推荐
- 重型系统

而是保持：

- 小而美
- 克制
- 安静
- 有留白
- 有作者气质

---

# 二、产品设计原则

## 1. 所有功能必须增强"仪式感"

新增功能必须服务于：

- 抽签感
- 命运感
- 等待感
- 落定感

而不是工具堆砌。

---

## 2. 保持"纸张感"和"静"

避免：

- 互联网平台风格
- 强社交 UI
- 游戏老虎机风格
- 炫技动画
- 强营销感

保持：

- 米纸感
- 木质感
- 留白
- 柔和动效
- 轻音效

---

## 3. 功能少，但完整

每个功能都必须：

- 明确
- 完整
- 自然
- 与整体气质一致

拒绝：

- 半成品功能
- "为了加而加"
- 信息密度过高

---

# 三、版本规划

| 版本 | 核心目标 |
|---|---|

# 四、v1.1 需求（体验完整性）

## 4.1 localStorage

### 背景

当前用户自定义菜单刷新后丢失。

这会导致：

- 用户投入感中断
- 产品不连续
- 体验不完整

因此需要本地持久化。

---

### 功能目标

实现：

- 用户菜单自动保存
- 页面刷新后恢复
- 用户无需登录
- 零打断体验

---

### 技术方案

采用：

```text
localStorage
```

不引入：

- 用户系统
- 微信登录
- 后端数据库
- OAuth
- 云同步

---

### 数据结构（v10）

统一 state（含版本号，便于未来迁移）：

```js
const state = {
  version: 10,
  foodMode: 'caixi',           // 当前菜品模式: shengji|caixi|jiachang
  myMenu: [],                  // 我的菜单
  selectedCategories: [],      // 已选中的分类
  excludedFoods: [],           // 从选中分类中排除的菜品
  categoryOrder: [],           // 分类排序（跨模式保留）
  customTips: {},              // 用户自定义贴士
  customRecipes: {},           // 用户自定义做法
  history: {
    recent: [],         // 最近吃过（FIFO, max 30）
    favorites: [],      // 收藏
    blacklist: [],      // 黑名单（随机时排除）
    todayFortune: null, // 今日签文缓存
    todayPick: null,    // 今日推荐缓存
    draws: []           // 完整抽签记录
  },
  settings: {
    sound: true
  }
}
```

---

### 存储 Key

```text
chisha-state
```

---

### 初始化逻辑

页面加载时：

1. 检查 localStorage
2. 若存在：恢复 state，执行 migrateState 迁移
3. 若不存在：使用默认状态
4. 根据 foodMode 加载对应菜品数据
5. 若当前选中分类不包含当前模式有效分类，重置为该模式默认

---

### 保存时机

以下操作后自动保存：

- 添加/删除菜品
- 选中/取消分类
- 排除/恢复菜品
- 分类排序变更
- 模式切换
- 随机结果生成
- 设置变更（如音效）

---

### 非目标

当前版本不做：

- 多设备同步
- 账户体系
- 云备份
- 分享状态同步

---

# 六、v1.3 需求（签文化）

## 6.1 食签系统

### 产品目标

增强产品灵魂感。

从：

> "随机吃什么"

升级为：

> "问天意"

---

### 设计原则

签文必须：

- 克制
- 有余味
- 有古意
- 不低俗

避免：

- 运势 App 风格
- 幸运数字
- 桃花运
- 网络鸡汤

---

### 签文类型

#### 1. 食签

示例：

```text
宜热食
忌久饿
```

---

#### 2. 节气签

示例：

```text
天寒，宜汤饼
```

---

#### 3. 微人生感

示例：

```text
饱腹之后，再谈远方
```

---

### 展示位置

结果卡下方。

要求：

- 小字号
- 不抢主结果
- 有留白

---

### 签文刷新机制

采用：

```text
方案 B：每日固定签
```

理由：

- 仪式感更强："今天的签是什么" > "每次不同"
- 社交传播：朋友之间可比较"今天你是什么签"
- 与分享图联动：每日一签分享图内容固定

实现：

基于日期的确定性随机（同一天所有用户看到相同签文）。

---

## 6.2 四柱干支日期

### 产品目标

增强中原文化气质。

展示当前公历日期对应的农历四柱（年柱、月柱、日柱、时柱），以传统干支纪年法呈现。

---

### 设计原则

- 精确：年柱以立春为界，月柱以节气为界
- 简洁：仅展示核心干支，不堆砌信息
- 安静：作为页面顶部的文化点缀，不抢主视觉

---

### 技术方案

内联 `calendar.js`（~69KB），支持 1900-2100 年农历/节气/干支计算。

VSOP87 节气模型 + 立春分界，与专业万年历对齐。

---

### 展示格式

```text
乙巳年 庚辰月 丁卯日 辛亥时
```

上下以细横线装饰，字体放大， serif 风格。

---

## 6.3 节气主题背景

### 产品目标

页面背景随当前节气微妙变化。

---

### 实现

24 节气对应 24 种米纸色值，通过 CSS 变量 `--paper` 动态切换。

例如：

- 立春：`#EEF0E0`
- 夏至：`#F0DEC5`
- 冬至：`#D6D3CD`

变化细微，用户可能察觉不到，但能感受到"时间在流动"。

---

# 七、v1.4 需求（菜品体系重构）

## 7.1 34 省经典名菜

### 产品目标

将原有的硬编码菜品体系升级为可扩展的 JSON 数据文件，按中国 34 个省级行政区组织，每个省 10 道经典名菜。

---

### 数据结构

`data/shengji.json`：

```json
{
  "categories": [
    {"name": "北京", "items": ["一品豆腐", "东来顺涮羊肉", ...]},
    {"name": "天津", "items": ["天津红烧牛尾", "火笃面筋", ...]},
    ...
  ],
  "tips": {
    "一品豆腐": "嫩如凝脂，入口即化",
    "东来顺涮羊肉": "铜锅炭火，羊肉薄如纸"
  }
}
```

共 34 个分类，340 个条目，339 道唯一菜品（"宫保鸡丁"在四川、贵州重复）。

---

### 每道菜配 tip

每条 tip 10-20 字，温暖、实用、有趣：

```text
"汤面一体，趁热吃更香"
"辛辣暖胃，配油条更佳"
```

---

## 7.2 菜品排除机制

### 产品目标

允许用户在选中分类的基础上，排除个别不想吃的菜。

---

### 实现

引入 `excludedFoods` 数组。签筒和老虎机最终菜品池 = 选中分类的所有菜品 - excludedFoods。

分类未选中时，该分类下的排除记录自动清空。

---

# 九、v1.6 需求（三模式切换）

## 9.1 产品目标

满足不同场景下的"今天吃什么"需求：

- 想吃传统中华名菜 → 按省籍或按菜系浏览
- 想吃日常便饭 → 按场景浏览

---

## 9.2 三种模式

| 模式 | 数据文件 | 分类逻辑 | 菜品数 | 默认选中 |
|---|---|---|---|---|
| 省籍 | `data/shengji.json` | 34 个省级行政区 | 340 道 | 北京 + 我的菜单 |
| 菜系 | `data/caixi.json` | 13 大传统菜系 | 372 道（去重后） | 河南菜 + 我的菜单 |
| 家常 | `data/jiachang.json` | 8 个日常场景 | 80 道 | 家常小炒 + 我的菜单 |

---

### 省籍模式

按中国 34 个省级行政区组织，每省 10 道经典名菜。

来源：《中国菜》名录及各地经典名菜整理。

---

### 菜系模式

按传统菜系组织：

河南菜（35）、川菜（35）、粤菜（35）、鲁菜（29）、苏菜（30）、浙菜（25）、闽菜（25）、湘菜（25）、徽菜（25）、东北菜（25）、西北菜（24）、西南菜-云贵（25）、京鄂沪（34）

总计 372 道（去重 2 道：葱油饼、臊子面）。

河南菜保留原有 35 条手工 tip，其余 337 道通过 AI 批量生成。

---

### 家常模式

按日常用餐场景组织：

- 家常小炒（10）
- 汤羹粥品（10）
- 面食主食（10）
- 外卖热门（10）
- 烧烤炸物（10）
- 清淡素食（10）
- 快手早餐（10）
- 周末硬菜（10）

总计 80 道，全部配 tip。

---

## 9.3 状态隔离

三种模式共享同一套 state，但通过分类名自然隔离：

- 省籍用「北京」「四川」等
- 菜系统用「河南菜」「川菜」等
- 日常用「家常小炒」「汤羹粥品」等

`selectedCategories`、`categoryOrder`、`excludedFoods` 跨模式保留，不互相覆盖。`getOrderedCategories()` 会自动过滤当前模式下不存在的分类名。

---

## 9.4 切换逻辑

按钮循环顺序：**省籍 → 菜系 → 家常 → 省籍**

点击后：

1. 重新加载对应 JSON
2. 过滤选中分类为当前模式有效分类
3. 若无有效分类，重置为该模式默认
4. 不清空 excludedFoods（跨模式保留，无效项自然无影响）
5. 刷新老虎机、选菜列表、签筒数量

---

## 9.5 默认行为

应用默认打开 **菜系** 模式，默认选中 **河南菜 + 我的菜单**。

切换到省籍/家常时，默认仅选中 **我的菜单**。
