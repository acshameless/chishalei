# 《吃啥嘞》产品需求文档（PRD）

## 项目名称

吃啥嘞（chisha.shameless.top）

一句话定义：

> 一个带有“抽签仪式感”的中原饮食随机选择器。

---

# 一、产品定位

## 产品目标

解决：

> “今天吃什么” 的决策疲劳。

但不仅仅是功能性解决。

产品核心价值：

- 提供轻量随机决策
- 提供情绪价值
- 提供仪式感
- 提供文化气质

本产品不追求：

- 功能复杂
- 平台化
- 社交化
- AI 推荐
- 重型系统

而是保持：

- 小而美
- 克制
- 安静
- 有留白
- 有作者气质

---

# 二、产品设计原则

## 1. 所有功能必须增强“仪式感”

新增功能必须服务于：

- 抽签感
- 命运感
- 等待感
- 落定感

而不是工具堆砌。

---

## 2. 保持“纸张感”和“静”

避免：

- 互联网平台风格
- 强社交 UI
- 游戏老虎机风格
- 炫技动画
- 强营销感

保持：

- 米纸感
- 木质感
- 留白
- 柔和动效
- 轻音效

---

## 3. 功能少，但完整

每个功能都必须：

- 明确
- 完整
- 自然
- 与整体气质一致

拒绝：

- 半成品功能
- “为了加而加”
- 信息密度过高

---

# 三、版本规划

| 版本 | 核心目标 |
|---|---|
| v1.1 | 体验完整性（localStorage + 音效） |
| v1.2 | 传播能力（分享图） |
| v1.3 | 产品灵魂（签文化） |

---

# 四、v1.1 需求（体验完整性）

# 4.1 localStorage

## 背景

当前用户自定义菜单刷新后丢失。

这会导致：

- 用户投入感中断
- 产品不连续
- 体验不完整

因此需要本地持久化。

---

## 功能目标

实现：

- 用户菜单自动保存
- 页面刷新后恢复
- 用户无需登录
- 零打断体验

---

## 技术方案

采用：

```text
localStorage
```

不引入：

- 用户系统
- 微信登录
- 后端数据库
- OAuth
- 云同步

---

## 数据结构

统一 state（含版本号，便于未来迁移）：

```js
const state = {
  version: 1,
  foods: [],
  history: {
    recent: [],         // 最近吃过（FIFO, max 30）
    favorites: [],      // 收藏
    blacklist: [],      // 黑名单（随机时排除）
    todayFortune: null, // 今日签文缓存
    todayPick: null     // 今日推荐缓存
  },
  settings: {
    sound: true
  }
}
```

---

## 存储 Key

```text
chisha-state
```

---

## 初始化逻辑

页面加载时：

1. 检查 localStorage
2. 若存在：恢复 state
3. 若不存在：使用默认菜单

---

## 保存时机

以下操作后自动保存：

- 添加菜品
- 删除菜品
- 随机结果生成
- 设置变更（如音效）

---

## 非目标

当前版本不做：

- 多设备同步
- 账户体系
- 云备份
- 分享状态同步

---

# 4.2 音效系统

## 产品目标

增强：

- 抽签仪式感
- 等待感
- 结果落定感

音效定位：

> “轻、木、纸、静”

---

## 设计原则

避免：

- 游戏老虎机感
- 高频循环音效
- 强提示音
- 电子感 UI 声音

保持：

- 短促
- 克制
- 留白
- 有余韵

---

## 音效列表

### 1. 开始抽签

触发时机：

点击“问问天意”开始转动。

音效选定：

- Freesound: Woodblock #268822 (kwahmah_02)
- 木块轻敲，干净短促，东方木质感
- 许可证：CC0

时长：

```text
0.2s - 0.4s
```

---

### 2. 结果落定

触发时机：

结果停止时。

音效选定：

- Freesound: Wax seal #759526 (Cerise_Virtuelle)
- 蜡封印章落纸，古意仪式感
- 许可证：CC0

时长：

```text
0.1s - 0.2s
```

---

## 技术方案

使用：

```text
Web Audio API
```

方案：

```js
const audioCtx = new (window.AudioContext || window.webkitAudioContext)();
```

优势：低延迟、移动端兼容性更好。

首次用户交互时解锁音频上下文（iOS Safari / 微信要求）。

---

### 功能

允许用户关闭声音。

---

## UI 方案

右下角小型图标：

```text
♪
```

点击切换：

- 开启
- 静音

要求：

- 不抢视觉
- 默认开启
- 状态持久化

# 五、v1.2 需求（传播能力）

## 产品目标

增强：

- 微信传播
- 小红书传播
- 截图分享欲

核心思想：

> 分享“签”，而不是分享链接。

```text

分享这道菜（分享图 A）
```

生成对应分享图

↓

用户保存图片（微信内长按保存）

↓

发送至：

- 微信
- 朋友圈
- 小红书
- 微博


- 微信 JS-SDK
- Puppeteer
- 后端图片服务

---

## 分享图视觉原则

分享图必须像：

> 一张签。

而不是营销海报。

---

## 分享图结构

推荐布局：

```text

胡辣汤

宜热食
忌久饿

```

---

## 视觉要求

保持与主站一致：

- 米纸背景
- 点阵纹理
- 漆红色
- serif 标题
- 大留白
- 极简布局

---

## 明确禁止

禁止：

- 巨大二维码
- 强营销文案
- “快来使用”
- 社交裂变风格
- 信息过载
- 花哨渐变

---

## 分享链接策略

当前阶段：

仅展示站点链接：

```text

```

- 可恢复随机结果

# 六、v1.3 需求（签文化）

# 6.1 食签系统

## 产品目标

增强产品灵魂感。

从：

> “随机吃什么”

升级为：

> “问天意”

---

## 设计原则

签文必须：

- 克制
- 有余味
- 有古意
- 不低俗

避免：

- 运势 App 风格
- 幸运数字
- 桃花运
- 网络鸡汤

---

## 签文类型

### 1. 食签

示例：

```text
宜热食
忌久饿
```

---

### 2. 节气签

示例：

```text
天寒，宜汤饼
```

---

### 3. 微人生感

示例：

```text
饱腹之后，再谈远方
```

---

## 展示位置

结果卡下方。

要求：

- 小字号
- 不抢主结果
- 有留白

---

## 签文刷新机制

采用：

```text
方案 B：每日固定签
```

理由：

- 仪式感更强：“今天的签是什么” > “每次不同”
- 社交传播：朋友之间可比较“今天你是什么签”
- 与分享图联动：每日一签分享图内容固定

实现：

基于日期的确定性随机（同一天所有用户看到相同签文）。
